## 2. GIT, VERSIONAMIENTO Y TRABAJO PROFESIONAL
### 4.  Explica con tus palabras el flujo básico de trabajo en Git (commit, push, pull, branch, merge).
El flujo básico de trabajo en Git empieza con crear cambios en los archivos del proyecto.
Cuando ya están listos, hago un commit, que es como guardar una versión de mi trabajo en mi computadora.
Después, uso push para enviar esos cambios al repositorio en GitHub.
Antes de empezar a trabajar, se recomienda hacer pull, que sirve para traer los cambios más recientes que hicieron otras personas del equipo, evitando conflictos.
Una branch es una rama donde puedo trabajar en nuevas funciones sin afectar el proyecto principal, y finalmente, cuando termino, se hace un merge, que une los cambios de mi rama con la rama principal o con otra rama del equipo.

### 5. ¿Por qué es importante usar control de versiones en proyectos reales?
Es importante porque permite llevar un historial completo de todos los cambios que se hacen en el proyecto.
Ayuda a trabajar en equipo de manera organizada, permite recuperar versiones anteriores en caso de errores y evita que se pierda trabajo.
Con control de versiones, todos pueden colaborar sin sobrescribir el trabajo de otros, y se mantiene un registro claro sobre qué se hizo, cuándo se hizo y quién lo hizo.

### 6. ¿Cómo aplicarías GitHub en un proyecto profesional o universitario en equipo?
Aplicaría GitHub creando un repositorio donde todo el equipo pueda colaborar.
Asignaría ramas específicas para diferentes tareas, usaría issues para organizar el trabajo, y cada miembro subiría sus avances mediante commits y pull requests.
Esto permite revisar el código antes de unirlo, mantener un flujo profesional y evitar errores.
Además, GitHub facilita almacenar documentación, controlar versiones y mantener comunicación clara sobre el progreso del proyecto.
